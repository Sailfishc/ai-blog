## Old Assumptions

**Old assumption 1: 产品经理的核心价值是协调长期路线图。**  
Why it used to be rational: 当代码昂贵、发布慢、跨团队依赖多时，提前几个月对齐路线图能减少返工。  
What changed in the materials: 功能规划周期被压到一周甚至一天，很多想法可以周末交给用户。  
New scarcity: 定义最小核心任务、排除无关解法、让团队马上行动的判断力。  
Reader decision that should change: 少问“下两个季度怎么排”，多问“这周什么目标清楚到足以发布一个可反馈版本”。

**Old assumption 2: PRD 是团队共识的中心。**  
Why it used to be rational: 文档是慢团队里最便宜的对齐媒介。  
What changed in the materials: AI 可以快速生成文档，但真正的产品差异来自目标、失败模式、eval 和核心用例。  
New scarcity: 判断哪些模糊性必须写清，哪些只需要单页目标，哪些必须转成 eval。  
Reader decision that should change: 把 PRD 从“规格说明书”降级为“目标和故障模式的快照”。

**Old assumption 3: 角色边界保证专业质量。**  
Why it used to be rational: 工程、设计、PM 各自工具链和交付成本高，边界能减少混乱。  
What changed in the materials: 工程师可以直接从用户反馈到周末发布，设计师也提交代码，PM 也写代码。  
New scarcity: 产品品味、一致性治理、跨职能取舍。  
Reader decision that should change: 不再用职位判断谁该行动，而用“谁最能定义正确结果”判断谁该负责。

**Old assumption 4: 产品功能越多，产品越强。**  
Why it used to be rational: 过去每个功能都贵，能上线通常说明它经过充分筛选。  
What changed in the materials: AI 让试验形态增多，模型升级又会让旧提示和旧功能变成拐杖。  
New scarcity: 删除能力、用户路径教育、产品一致性。  
Reader decision that should change: 每次模型或能力升级后，专门审查哪些辅助层应该弱化或删除。

**Old assumption 5: 自动化达到 90% 就已经很有价值。**  
Why it used to be rational: 人工流程昂贵，半自动也能省时间。  
What changed in the materials: AI 原型很容易，但最后 5-10% 的可靠性决定它是否真能托管。  
New scarcity: 反馈训练、偏好沉淀、可复现失败案例。  
Reader decision that should change: 只把少数高频任务推进到可信自动化，不沉迷无限定制。

## Article

# Anthropic 产品团队快的不是执行，而是把发布变成了作业场

Cat Wu 在 Lenny 的访谈里讲了两个看起来不相干的场景。

第一个场景很像今天所有 AI 产品团队的日常。周日晚上冒出一个最高优先级任务，到了周一变成更高优先级，周一下午又继续升级。她说，很多时候你回头看，甚至会觉得自己不敢相信曾经为周日那个 P0 那么焦虑。

第二个场景更具体。Claude Code 的工程师觉得一个功能准备好了，就把它放到内部的常态化发布频道里。文档、产品营销、开发者关系团队看到以后，最快第二天就能完成公告、文档和对外说明。功能本身往往以 research preview 发出去，明确告诉用户：这是早期想法，我们希望获得反馈，也不承诺长期支持。

很多人看到这种公司，会本能地得出一个简单解释：他们快，是因为他们有最好的模型。

这个解释太省事了。

访谈里真正值得看的，不是“Anthropic 用 AI 所以快”，而是它把“快”拆成了一组可重复的作业面：目标如何被压缩，谁可以行动，发布摩擦如何被清掉，反馈如何进入下一轮，哪些旧功能应该被删除，哪些自动化必须做到接近可信。

换句话说，Anthropic 产品团队快的不是执行本身。

它快，是因为它把发布变成了一个作业场。

## 旧价格表：产品经理负责让未来稳定

在 AI 之前，产品管理的旧假设很合理。

代码贵，协调贵，发布慢。一个功能可能要跨越多个季度，牵涉多个合作团队。如果路线图不提前对齐，工程做到一半，另一个团队的变化可能让你的计划失效。于是好的产品经理很大一部分价值，是写清楚长期计划、协调依赖、管理节奏、避免返工。

这不是官僚主义。它是旧价格表下的理性选择。

当实现昂贵时，最怕的是做错；当发布稀疏时，最怕的是错过窗口；当团队边界稳定时，最怕的是没人提前沟通。

Cat Wu 的判断是，这张价格表已经被改写了。

她说，很多产品功能的规划时间，从过去的 6 到 12 个月，缩短到一个月、一周，甚至一天。AI 提升了工程效率，模型能力本身也在快速变化。一个想法不再天然需要几个季度的路线图，它可能只需要一个足够清楚的目标、一个愿意动手的人、一套低摩擦发布流程。

所以产品经理的中心任务开始移动。

不是“如何确保多季度路线图完美对齐”，而是“如何让一个工程师或产品经理周末想到的东西，尽快以可反馈的形态交到用户手上”。

这句话听起来像速度崇拜，但它其实不是。

因为它隐含了一个更难的问题：如果每个人都能更快行动，什么东西会变稀缺？

答案不是代码。

答案是目标、品味、验证、发布通道和删除旧东西的勇气。

## 新瓶颈：不是做得出，而是知道做什么

Cat Wu 多次回到“产品品味”这个词。

这个词很容易被误读成审美、直觉或高级感。但在这次访谈里，它不是一个软词。它是一个成本结构词。

当写代码的成本下降，最值钱的就不再是“能不能写出来”，而是“该不该写、写成什么样、为谁写、用什么路径让用户成功”。

她举的目标定义例子很具体：Claude Code 的核心用户是企业里的专业开发者；问题是权限提示过多造成疲劳；目标用例是让他们安全地实现零权限提示。

这个目标句的价值，不在于它漂亮，而在于它排除了很多看似相关的方案。减少权限提示有很多办法，但“企业专业开发者”“安全”“零权限提示”这几个约束放在一起，会强迫团队放弃一批浅层优化，转向真正能让核心用户成功的路径。

这就是 AI 产品里产品经理的新工作：把通用模型带来的巨大可能性，压缩成一个团队今天能行动、明天能验证、下周能发布的判断。

在旧世界里，模糊性常常靠长文档吸收。在新世界里，模糊性要么被压缩成清晰目标，要么被转成 eval，要么被 research preview 暴露给用户。

这也是为什么 PRD 没有消失，但位置变了。

Cat Wu 说，对于特别模糊的功能，单页文档仍然有帮助：目标是什么，令人愉悦的用例是什么，当前需要解决的故障模式是什么。对于重大基础设施项目，仍然可能需要几个月，也仍然会写 PRD。

但 PRD 不再是产品管理的宗教中心。

更关键的是：团队是否每周理解指标，是否共享核心用户和取舍原则，是否知道哪些事可以自己决定，是否能用小而准的 eval 定义成功。

她提到，10 个优秀 eval 就足以帮助团队量化目标、进展和缺口。这个判断很重要。因为 eval 在这里承担的不是测试工程师的边角工作，而是产品定义的一部分。

过去，PM 说“这个体验应该更好”。现在，PM 需要能说：“这是 5 个评估，这是运行方式，这是成功样例，这是失败样例，这是我用来提高成功率的提示词或产品约束。”

产品品味开始变成一种可运行的判断系统。

## 发布作业场：把速度拆成五个作业面

Anthropic 的速度不是一个单点技巧。它更像一个作业场，至少包含五个作业面。

第一，目标作业面。

目标必须足够窄，窄到它能排除错误方案。AI 原生产品最大的问题不是没有想法，而是想法太多。通用模型什么都能做一点，于是团队很容易同时追逐十几个方向。产品经理的工作，是把用户、问题、核心用例、非目标用户写清楚，让团队能不等审批就做局部决策。

第二，发布作业面。

research preview 是一个被低估的组织机制。它不是给不成熟产品找借口，而是在重新定价“发布承诺”。如果每个发布都被当成长期稳定承诺，团队自然会慢下来；如果早期功能可以被清楚标记成研究预览，团队就能更快拿到真实反馈。

但这要求组织诚实。你必须告诉用户它是什么，也必须真的接住反馈。

第三，跨职能作业面。

Cat Wu 描述的常态化发布频道，本质上是在把跨职能协作模板化。工程师不需要每次临时找 PMM、文档、DevRel 开一轮会。功能就绪、内部文档写好、丢进发布频道，负责文档的 Sarah、负责产品营销的 Alex、DevRel 的 Tarek 和 Lydia 就能按既定节奏接上。

这类流程听上去普通，但它解释了很多速度差异。

很多团队不是慢在写代码，而是慢在“写完以后不知道谁该介入”。AI 把实现变快以后，这种发布摩擦会被放大。以前一周的工程工作后面跟两天协调，不明显；现在一天的工程工作后面跟两天协调，就会变成主要瓶颈。

第四，验证作业面。

Anthropic 官方 Code Review 发布文给了一个很好的外部校验：他们说 Anthropic 工程师的代码产出一年增长了 200%，code review 变成瓶颈；Code Review 上线后，有实质审查评论的 PR 从 16% 提升到 54%。这个工具会派出多个 agent 做深度审查，但官方也明确说，它不会批准 PR，最终批准仍然是人的判断。

这很关键。

AI 让输出增加以后，验证不会自动减少，反而会变成拥堵点。真正成熟的团队不会假装“AI 写的就让 AI 过”，而是把验证也产品化：多 agent 审查、严重程度排序、误报过滤、成本控制、最后由人批准。

第五，删除作业面。

这是 AI 产品最反直觉的一层。

Cat Wu 讲了 Claude Code 的待办列表。早期模型做大型重构时，会说自己要改 20 个调用点，最后改 5 个就停。团队给模型加了待办列表，让它像人类一样追踪所有调用点。后来模型变强，不再需要被强制使用这个列表。于是待办列表从“模型完成任务的必要拐杖”，变成“用户查看进度的 UI 元素”。

她还说，每次新模型发布，团队都会通读系统提示词，问每一节是否还需要。如果不需要，就移除。

AI 产品不是只会加功能。AI 产品还必须会删掉昨天为弱模型搭的脚手架。

这就是发布作业场和普通发布流程的区别：它不仅把东西推出去，也把过期东西收回来。

## 为什么“全员跨界”会同时提高速度和制造坏账

访谈里有一个很容易被鸡血化的部分：所有角色都在融合。

产品经理写代码，工程师做产品管理，设计师也提交代码。Claude Code 团队里很多工程师能从 Twitter 用户反馈出发，到周末直接发布功能，几乎不需要 PM 介入。Cat Wu 认为，最理想的工程师是有强产品品味的工程师。

这当然会快。

但她没有把代价藏起来。她说，团队牺牲的是产品一致性。

过去写代码贵，团队会仔细规划产品套件中每个功能的关系，尽量做到每个用例对应清晰产品。现在，想法太多，测试太快，内部也可能同时喜欢两种形态，于是外部用户会看到功能重叠、路径分叉、同一个问题有不同表现形式。

这会给用户带来成本：他们不知道完成任务的最佳路径是什么。

所以 Claude Code 后来做了 slash power up 一类功能。Cat Wu 说，团队原本不想做 onboarding，因为他们相信产品应该足够直观。但功能实在太多，用户开始问：你们有 100 项功能，我到底必须掌握哪 10 个？

这就是速度的坏账。

越快的团队，越需要给用户做路径压缩。否则用户不是被你赋能，而是被你追着跑。

这也解释了为什么“产品品味”在这个阶段更重要，而不是更不重要。AI 可以让更多人提交代码，但它不能自然保证产品组合仍然清楚。角色边界消融以后，团队需要另一种约束取代旧边界：共同使命、团队原则、核心用户定义、功能删除机制、用户教育路径。

否则跨界只是混乱的好听说法。

## Cowork 的例子：AI 省下的是制稿时间，不是判断时间

Cat Wu 讲 Cowork 的例子很适合校准大家对知识工作自动化的想象。

她准备 Code with Claude 大会演讲，要讲 Claude Code 如何从助手转成完整 agent。她把 Google Calendar、Slack、Gmail、Google Drive 都接给 Cowork，又提供了产品营销经理 Alex 建议覆盖的要点、自己不满意的草稿和想讲的故事线。

Cowork 跑了一个小时。它浏览 Twitter，看发布了什么；看常青发布室；看 Claude Code 公告频道；整合团队如何使用 Claude Code 的演示，最后生成 20 页演示文稿。第二天早上她读了一遍，觉得相当不错，但文字太长，还要自己调整。

这个例子不是“AI 替 PM 做完了演讲”。

恰恰相反，它显示了 PM 判断的位置移动了。

AI 做了大量过去很耗时的综合：找材料、读上下文、按设计系统生成草稿、把内部案例组织成页面。人做的是最终故事线：从本地任务成功，到每个 PR 显示通过，再到帮助工程师合并更多 PR；哪些 demo 最有说服力；哪些内容不要和主 keynote 重叠；哪些文字要删短。

这和她对 Claude Code / Cowork 的工具路由一致：代码输出用 Claude Code，非代码输出用 Cowork；终端适合一次性强任务，桌面适合前端预览，移动端适合随时启动任务。

未来不是一个聊天框吞掉所有工作。至少在当前阶段，更现实的能力是任务路由：什么任务该进代码 agent，什么任务该进知识工作 agent，什么任务需要图形预览，什么任务只能由人做最后取舍。

AI 把制稿时间打下来以后，故事线判断反而更显眼。

## 内部工具爆发以后，真正的问题是维护边界

访谈里还有一个被很多人低估的细节：Claude Code 让定制办公软件的门槛大幅下降。

Cat Wu 说，公司里出现了更多按具体用例构建的软件。一个销售同事发现自己一直重复制作类似演示文稿，于是用 Claude Code 做了一个 Web app。它可以输入特定客户信息，从 Salesforce、Gong 和其他笔记里调取数据，判断客户使用 Bedrock、Claude for Enterprise 还是 Console；如果客户关心 LLM 代码审查，就加代码审查页；如果客户需要 HIPAA 或安全控制，就加相关页面；如果客户用 Vertex 或 Bedrock 且不想用 Claude for Enterprise，就删掉只适用于 Claude for Enterprise 的幻灯片。

过去这类定制可能需要 20 到 30 分钟，很多人要么做，要么直接放弃用通用模板。现在几秒钟就能生成。

这就是 AI 对 SaaS 的真实冲击：不是所有通用软件都会立刻消失，而是大量夹在“太小不值得买软件”和“太具体无法用通用 SaaS 解决”之间的内部场景，突然值得做了。

但这也会制造新问题。

当每个人都能做内部工具，谁负责维护？数据权限怎么控？客户承诺谁把关？工具什么时候淘汰？同一个问题会不会出现五个半成品？

所以发布作业场不能只覆盖外部产品，也要覆盖内部工具。工具变便宜以后，治理不应该消失，而应该变轻、变明确：什么场景允许自建，什么数据源可以接，谁是 owner，什么时候归档，什么时候升级成正式系统。

否则内部工具会从杠杆变成新一轮噪音。

## 自动化的分水岭不是 90%，而是最后 10%

访谈末尾，Cat Wu 给普通知识工作者的建议非常实际：找到工作里重复、讨厌、手动的部分，用 Claude Code、Cowork 或其他 AI 工具自动化。这样你可以把时间放回创造性工作。

但她紧接着补了一刀：很多人做到 90-95% 就放弃了，而自动化如果不能 100% 成功，就不算真正的自动化。

这句话容易被误解。开放世界里当然很难保证绝对 100%。她真正强调的是：只差 5% 的自动化，在很多场景里仍然会持续占用你的注意力。比如邮件分类 95% 正确，但偶尔把重要邮件扔进垃圾文件夹，你就仍然必须检查它；一旦你必须检查，它就没有真正释放你。

最后 5-10% 不靠再写一个酷 prompt。它靠偏好训练、反馈、可复现失败案例、技能更新、人工检查点和长时间迭代。

这也解释了她为什么反对沉迷无限定制。有人从不定制，错过杠杆；另一类人疯狂添加技能、MCP、工作流，甚至偏离发布产品或完成任务的核心目标。她的判断是：简单配置往往更好。

这不是保守，而是 ROI。

AI 自动化的目标不是让你的配置看起来强，而是让某个真实高频任务从“我每次都要做”变成“我可以信任它完成”。如果达不到这个标准，就不要把自己困在工具装修里。

## 这套机制什么时候会失败

发布作业场不是万能药。它至少有四个危险区。

第一，research preview 被滥用。

早期发布可以降低承诺成本，但不能掩盖安全、隐私、合规和核心工作流风险。对企业开发者、客户数据、权限系统、代码审查这类场景，preview 仍然需要明确边界。快不是把风险转嫁给用户。

第二，产品一致性被速度吃掉。

Cat Wu 已经点出这个代价。功能重叠、新手路径不清、用户每天被迫看更新，都会让产品从“强大”变成“难跟”。如果团队只奖励发布数量，不奖励删除、整合和教育，速度会变成体验债。

第三，eval 变成新的幻觉。

10 个好 eval 很有用，但 10 个差 eval 会制造假安全。AI 产品团队必须持续问：这些评估是否真的代表核心任务？是否覆盖边缘案例？是否被模型过拟合？是否只是把 PM 的偏见自动化？

第四，人类责任被自动化稀释。

官方 Code Review 文档里最重要的一点之一，是它不会批准 PR。它可以派多个 agent 深读、排序、过滤误报，但最终是否合并仍是人的决定。这个边界必须保留。AI 可以扩大审查覆盖面，但不能替团队承担责任。

## 给团队的决策框架

如果把这篇访谈压成一个可执行框架，可以用下面这张表。

| 场景 | 不该再执着 | 应该建设 |
|---|---|---|
| 需求很多 | 多季度路线图的完整感 | 一句话核心目标、非目标用户、核心失败模式 |
| 功能早期不确定 | 完美发布承诺 | research preview、反馈频道、退出条件 |
| 工程速度上升 | 更多临时协调会议 | 常态化发布频道、文档/PMM/DevRel 接力标准 |
| 文档更易生成 | 长 PRD 崇拜 | 单页目标、故障模式、eval、决策记录 |
| 角色边界变模糊 | 用职位阻止行动 | 用产品品味、核心用户和质量门约束行动 |
| 内部工具变便宜 | 全员乱造工具 | owner、数据边界、归档规则、升级路径 |
| 自动化达到 90% | 觉得已经完成 | 继续补最后 10% 的偏好、反馈和失败复现 |
| 模型升级 | 只加新功能 | 审查旧提示、旧拐杖、旧 onboarding 是否该删除 |
| 代码输出暴涨 | 让 review 自然跟上 | 多 agent 审查、人类批准、成本和误报控制 |
| 用户跟不上 | 继续堆功能 | power up、核心路径教育、功能合并和删减 |

对个人来说，最重要的问题也变了。

过去你可以问：我会不会写？我是不是这个职位？这个任务是不是归我？

现在更好的问题是：我是否理解限制条件？我是否知道成功是什么？我能不能把任务交给 AI 做掉 70%，然后把自己的判断放在最关键的 30%？如果这个自动化值得做，我是否愿意把它从 90% 打磨到可信？

对团队来说，问题也变了。

不是“我们能不能更快写代码”，而是“我们有没有一个发布作业场，能把更快的代码变成更快的学习，而不是更快的混乱”。

Anthropic 的经验最有价值的地方，不是它证明了某个 AI 工具多强。工具会变，模型会变，今天的拐杖明天可能被删除。

真正可迁移的是这句话：

当执行变便宜，组织的优势不再是执行更多，而是更快把执行接入目标、验证、发布和删除。
