# Target Style Pattern

## Core formula

```text
Insight = timely material × old assumption shift × cost-structure revaluation × portable mechanism × actionable decision
```

The target is a high-insight Chinese technical/business essay. It should preserve the writer's own voice while using the same argument mechanics.

## Article chain

```text
Concrete scene or event
→ Reader's old intuition
→ Why the old intuition used to work
→ New cost structure
→ New bottleneck or scarcity
→ Named mechanism
→ Multi-case validation
→ Boundary cases
→ Decision framework
→ Compressed ending
```

## What creates the “insight” feeling

### 1. Re-price the world

Ask what became cheaper, expensive, risky, abundant, scarce, faster, slower, easier to fake, or harder to verify.

Examples:

- Code generation becomes cheaper, feedback and review become scarce.
- Multi-source consensus becomes easier to fake, original-source tracing becomes scarce.
- Skill sharing becomes easy, personal loading boundaries become scarce.

### 2. Make the old assumption sympathetic

A strong article shows why the old way made sense under the old price table. This prevents cheap contrarianism.

Template:

```text
The old workflow existed because [old cost] was high. Once [new capability] lowers that cost, the work surface moves to [new bottleneck].
```

### 3. Name the mechanism

A mechanism name lets readers carry the argument away. Good names are short and decomposable.

Patterns:

- New infrastructure: 反馈基础设施, Context Infrastructure, 任务派发层
- New scarcity: 品味成本, 原始现场, 关系与此刻
- New operating model: 白班/夜班, 共享池/个人 INDEX, 学习回路
- New constraint: 锯齿边界, 三角约束, 自杀基因

### 4. Validate across cases

Use multiple concrete examples to show the same mechanism recurring. A single source can still provide multiple cases if it contains several workflows, failures, or decisions.

### 5. End with a decision frame

The ending should change the reader's next move:

- What should they build?
- What should they stop over-optimizing?
- What should they measure?
- What should they route to a different person/model/tool?
- What should they make explicit in process or architecture?

## Preferred section types

- Two failure scenes
- New price table
- Three-part mechanism breakdown
- Boundary section
- Decision matrix
- Role/team implications
- Final portable sentence

## Language notes

- Lead with concrete nouns and actions.
- Use short paragraphs.
- Let tables carry comparisons.
- Keep examples close to claims.
- Convert abstract labels into operational phrases.
- Prefer “这会改变 X 的判断方式” over broad claims like “这很重要”.
